# RFZv5 架构设计

> 从 v45 拆分而来。v45 是单体多租户（46 表 / 6 角色），v5 拆成独立系统，各司其职。

## 为什么拆分

v45 的问题不是代码量，是**角色 × 表 × 权限的笛卡尔积复杂度**。所有角色挤在一套菜单里，测试和设计都像意大利面。

拆分原则：**每套系统只服务一个场景，角色简单，菜单扁平。**

## 三个系统（我们只先做两个）

```
┌──────────────────────────────────────────────────────┐
│                    如法造 (Rufazao)                   │
│  产品主数据 · 供应链 · CRM · 订单履约                │
│                                                      │
│  角色：平台管理员 / 运营 / 产品维护                  │
└──────────┬───────────────────────────────┬───────────┘
           │ 产品同步 ↓       订单回传 ↑  │
           │                               │
┌──────────┴──────────┐         ┌──────────┴──────────┐
│   流通处A (独立域名)  │         │   流通处B (独立域名)  │
│   H5 在线商城        │         │   H5 在线商城        │
│   独立 DB · 独立管理  │         │   独立 DB · 独立管理  │
│                      │         │                      │
│   角色：管理员 / 店员 │         │   角色：管理员 / 店员 │
└─────────────────────┘         └─────────────────────┘

  机构系统（后续再设计）
```

## 流通处 — H5 在线商城

### 定位

面向信众消费者的法物流通零售终端。独立域名，独立品牌，独立管理。

### 能力

| 能力 | 说明 |
|------|------|
| 导入如法造产品 | 从如法造拉取产品目录，作为商品基础数据 |
| 自己定价 | 如法造给基准价，流通处自行定价 |
| 自己上下架 | 决定哪些产品在自己的商城可见 |
| 库存管理 | 自己盘库存，做盘点 |
| 销售订单 | 面向消费者的零售订单 |
| 会员/客户管理 | 自己的顾客数据 |
| 补货申请 | 向如法造发起补货 |
| 订单回传如法造 | 订单数据同步给如法造做履约 |

### 数据边界

**流通处自己的数据（不跟如法造共享）：**
- 会员/客户（customer）
- 定价（pricing）
- 上下架状态
- 销售订单（sales_order）
- 库存（store_inventory）
- 盘点（inventory_count）

**从如法造同步过来的数据（只读）：**
- 产品 SPU（product）
- 产品 SKU（product_variant）
- 产品类目（product_category）

## 如法造 — 平台后台

### 定位

产品主数据中心 + 供应链管理 + CRM + 订单履约。

### 能力

| 能力 | 说明 |
|------|------|
| 产品管理 | SPU/SKU/类目，唯一真相源 |
| 供应链 | 采购、入库、出库 |
| CRM | 客户、联系人、沟通记录 |
| 订单履约 | 接收各流通处订单，统一处理 |
| 推送产品 | 产品变更推送给流通处 |
| API Key 管理 | 为流通处/机构签发 API Key |

### 不做的事

- ❌ 不直接面向消费者（没有 H5 商城）
- ❌ 不管流通处的定价/上下架
- ❌ 不管流通处的库存

## 同步协议

### 如法造 → 流通处（产品同步）

```
触发：产品新增/更新/下架
方式：Push（如法造主动调用流通处 API）
认证：API Key
数据：product + product_variant + product_category
频率：实时（变更即推）
```

### 流通处 → 如法造（订单回传）

```
触发：流通处产生新订单
方式：Push（流通处主动调用如法造 API）
认证：API Key
数据：sales_order + order_item
频率：实时（订单产生即传）
```

## 技术架构（继承 v45）

每套系统 = 独立的前端 + 独立的 DB + 独立的 Agent

```
流通处实例
├── admin-react/          # 流通处管理后台（继承 v45 SchemaTable/DetailPanel）
├── h5/                   # 消费者端 H5 商城（新增）
├── schema/               # .surql — 只包含流通处用到的表
├── server/               # Agent（处理聊天查询）
├── deploy.sh             # 部署脚本
└── config.yaml           # API Key / SDB 连接 / 域名
```

### 继承 v45 的成功模式

1. **Schema-driven UI** — SchemaTable / DetailPanel 从 .surql 自动生成
2. **ChatPanel + Agent** — 两段式 SQL 生成 → 执行 → 格式化
3. **菜单从注释驱动** — `-- @label` / `-- @group`
4. **FIELD_ZH 映射** — 所有字段名中文显示
5. **配置外置** — LLM prompt / API Key 独立于代码
6. **单一真相源** — .surql 是唯一的 schema 定义

### 抛弃 v45 的包袱

- 6 种角色权限判断
- 复杂的跨表 FETCH
- tenant 字段（流通处不需要多租户，每个实例自己就是租户）
- 多余的 30+ 张表（只保留流通处需要的 10 张左右）

## 第一步：流通处最小闭环

```
用户打开 H5 商城 → 浏览产品 → 下单
                         ↓
              流通处管理后台可见订单
                         ↓
              订单同步给如法造
```

三张核心表：
- `product` / `product_variant` — 从如法造导入
- `sales_order` / `order_item` — 流通处产生

API 同步先不做——先用**手动导入**验证模型正确，再自动化。
